# Инструкция по работе с сервером и клиентом СУБД

Данный документ содержит исчерпывающее руководство по архитектуре, настройке и глубокой технической реализации клиент-серверной СУБД.

---

## 1. Общая архитектура системы

СУБД реализована на языке C++ и представляет собой многопоточное клиент-серверное приложение, взаимодействующее по протоколу HTTP.

### Основные компоненты:
*   **Crow Web Server**: Используется как высокопроизводительный HTTP-движок.
*   **SQL Parser**: Самописный лексический и синтаксический анализатор для обработки SQL-запросов.
*   **DBMS Engine**: Ядро СУБД, отвечающее за работу с файловой системой (B-деревья, индексы, кэширование).
*   **AuthManager**: Подсистема безопасности, обеспечивающая аутентификацию (JWT) и авторизацию (RBAC).
*   **Heartbeat Monitor**: Система мониторинга распределенных узлов.
*   **Logger & Telemetry**: Системы журналирования транзакций и сбора метрик производительности.

## 2. Работа с сервером

### Запуск
Для запуска сервера выполните:
```bash
./build/course_server [порт]
```
По умолчанию сервер слушает порт **8080**. При старте он автоматически инициализирует структуру папок в директории `data` (если они отсутствуют).

### API Эндпоинты (Техническая спецификация)

1.  **`GET /health`**
    *   **Назначение**: Быстрая проверка работоспособности.
    *   **Ответ**: `{"ok":true,"result":"ok","error":""}`.

2.  **`POST /register`**
    *   **Тело (JSON)**: `{"user": "...", "password": "...", "role": "..."}`.
    *   **Особенности**: Роль может быть `reader`, `writer` или `admin`. Если роль не указана, назначается `reader`. Назначение ролей `writer` и `admin` требует прав текущего администратора (передача токена в заголовке).

3.  **`POST /login`**
    *   **Тело (JSON)**: `{"user": "...", "password": "..."}`.
    *   **Результат**: Возвращает JWT-токен в поле `result`.

4.  **`POST /query`**
    *   **Тело**: Обычный текст SQL-запроса (или несколько запросов через `;`).
    *   **Заголовки**: Требует `Authorization: Bearer <TOKEN>`.

5.  **`GET /metrics`**
    *   **Назначение**: Выгрузка данных телеметрии (среднее время запроса, процент ошибок).

6.  **`GET /heartbeat`**
    *   **Назначение**: JSON-список статусов всех узлов, указанных в `nodes.txt`.


### Проверка сервера
```
curl http://127.0.0.1:8080/health
```

## 3. Работа с клиентом

Клиент (`course_client`) — это утилита, которая инкапсулирует логику взаимодействия с HTTP API, позволяя работать с сервером как с локальной консолью.

### Порядок работы клиента:
1.  **Авторизация**: Клиент первым делом обращается к `/login` с указанными учетными данными.
2.  **Чтение SQL**: Принимает либо путь к файлу, либо строку с запросом.
3.  **Отправка**: Формирует HTTP POST запрос к `/query`, вставляя полученный токен в заголовки.
4.  **Обработка**: Очищает HTTP-заголовки ответа и выводит только тело (результат выполнения SQL).

### Пример запуска:
```bash
./build/course_client 127.0.0.1 8080 "SELECT * FROM users;" admin admin
```

## 4. Глубокая техническая реализация авторизации

### Хранение данных пользователей (Protobuf)
Для обеспечения безопасности и переносимости список пользователей хранится в файле `data/_system/users.pb`.
*   Используется библиотека **Google Protocol Buffers** (структуры `ProtoAuthUser` и `ProtoAuthFile`).
*   При старте `AuthManager` десериализует этот файл в `std::map<std::string, AuthUser>`.

### Безопасность паролей
1.  **Соль**: Для каждого пользователя создается уникальная соль (на базе его имени и секретной константы).
2.  **Хэширование**: Применяется `std::hash` (или специализированный крипто-хэш) от комбинации `salt + ":" + password`.
3.  **Верификация**: При логине сервер заново вычисляет хэш присланного пароля с солью из БД и сравнивает результаты.

### Механизм JWT (JSON Web Token)
Реализован вручную в `AuthManager` для полного контроля над процессом:
*   **Формат**: `base64(header).base64(payload).base64(signature)`.
*   **Подпись**: Используется **HMAC-SHA256** через библиотеку OpenSSL (`EVP_sha256()`).
*   **Секретный ключ**: Хранится внутри сервера. При изменении ключа все ранее выданные токены станут невалидными.
*   **Payload**: Содержит имя пользователя (`sub`), роль (`role`) и время истечения (`exp`).

### Система ролей (RBAC)
Права проверяются в методе `AuthManager::canExecute(role, statement)` перед выполнением каждой команды:
-   **reader**: Доступны только `SELECT` и `USE`.
-   **writer**: Доступны `SELECT`, `INSERT`, `UPDATE`, `DELETE`, `USE`.
-   **admin**: Доступны все операции, включая DDL (`CREATE`, `DROP`).

## 5. Механизм Heartbeat и nodes.txt

Система мониторинга позволяет серверу "знать" о состоянии всей инфраструктуры.

### Конфигурация (nodes.txt)
Файл должен иметь формат:
```text
<IP> <PORT> [команда_перезапуска]
127.0.0.1 8081 ./scripts/restart_node_1.sh
192.168.1.50 8080
```

### Принцип работы:
1.  **Фоновый поток**: Сервер запускает `std::thread`, который работает асинхронно.
2.  **Пинг**: Каждые 5 секунд поток обходит все узлы и пытается выполнить `GET /health`.
3.  **Восстановление**: Если узел недоступен, а в `nodes.txt` была указана `restartCommand`, сервер пытается выполнить эту команду через `std::system()`.
4.  **Статус**: Текущий статус (`alive: true/false`) сохраняется в памяти и доступен через API.

## 6. Технические особенности реализации сервера

### Многопоточность и Crow
Использование Crow позволяет обрабатывать запросы параллельно. Однако это создает риск "состояния гонки" (race condition) при доступе к данным.

### Блокировки (TableLocks)
Для решения проблемы конкурентного доступа реализована система мелкозернистых блокировок:
*   В классе `TableLocks` используется `std::unordered_map<std::string, std::shared_ptr<std::mutex>>`.
*   При обращении к таблице (например, `users`), сервер захватывает мьютекс именно для этой таблицы.
*   Это позволяет одному потоку записывать в `table1`, пока другой поток одновременно читает из `table2`.

### Телеметрия
Класс `Telemetry` собирает метрики в атомарные переменные (`std::atomic`).
*   **latency**: Время от получения запроса Crow до формирования ответа.
*   **error_rate**: Соотношение запросов, завершившихся системной или SQL-ошибкой.

## 7. Примеры работы

### Сценарий: Создание пользователя и работа от его имени

1.  **Регистрация через SQL**:
    ```bash
    ./build/course_client 127.0.0.1 8080 "REGISTER USER worker PASSWORD 'pass123' ROLE 'writer';" admin admin
    ```

2.  **Выполнение запроса от имени нового пользователя**:
    ```bash
    ./build/course_client 127.0.0.1 8080 "INSERT INTO my_table (id) VALUE (10);" worker pass123
    ```

## Поддерживаемые SQL команды

- **REGISTER USER**
    - Регистрирует нового пользователя системы.
    - Синтаксис: `REGISTER USER username PASSWORD 'password' [ROLE 'role'];`
- **CREATE DATABASE**
    - Создает новую базу данных.
- **DROP DATABASE**
    - Удаляет базу данных.
- **USE**
    - Выбирает активную базу данных.
- **CREATE TABLE**
    - Создает таблицу.
- **DROP TABLE**
    - Удаляет таблицу.
- **INSERT INTO**
    - Добавляет данные.
- **UPDATE**
    - Изменяет данные.
- **DELETE FROM**
    - Удаляет данные.
- **SELECT**
    - Извлекает данные.
